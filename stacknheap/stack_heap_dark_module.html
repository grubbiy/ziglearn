Core idea
What is the stack? What is the heap? Why do you care?
Concept
Memory in a program

When a program runs, it gets a piece of RAM. Inside that RAM, there are different areas. Two important ones:

  • Stack ‚Äì short-lived data while functions run.
  • Heap ‚Äì flexible storage for long-lived or bigger data.
The stack

Think of a stack of plates:

  • You put new plates on top.
  • You always remove from the top.
  • Last plate in = first plate out (LIFO).

The program uses the stack for things like:

  • Function inputs (parameters).
  • Local variables.
  • Return address (where to come back).
The heap

The heap is a big pool of memory that the program can grab from when it needs more space.

  • You ask for memory (allocate).
  • You get back an address (a pointer).
  • You must remember to give it back (free).

Good for:

  • Data that lives longer than one function.
  • Big or flexible size data (lists, strings, etc.).
Rule of thumb: if the data is small and dies with the function ‚Üí stack. If it is big or must live longer ‚Üí heap.
Code examples
How stack and heap look in simple C-style code.
Code
Stack example

Here everything lives on the stack. When the function ends, its local variables are removed automatically.

Local variables on the stack Stack 
    int add(int a, int b) {
        int sum = a + b;   // sum lives on the stack
        return sum;        // a, b, sum are removed after return
    }

    int main(void) {
        int result = add(2, 3); // result lives on the stack in main
        return 0;
    }
Stack overflow idea

If you keep adding more and more stack frames (for example with recursion), you can run out of stack. The program then crashes with a stack overflow.

Too deep recursion Stack 
    void recurse(int n) {
        int numbers[1000];   // large array on the stack
        numbers[0] = n;

        recurse(n + 1);      // calls itself again and again
    }

    int main(void) {
        recurse(0);          // eventually: stack overflow (crash)
        return 0;
    }
Takeaway: large local arrays + deep recursion = easy way to kill the stack.
Heap example

Here we ask for memory on the heap and later give it back. The arr pointer itself is on the stack, but it points into the heap.

Basic heap allocation Heap 
    #include <stdlib.h>

    void make_array(int count) {
        int *arr = malloc(count * sizeof(int)); // memory from heap
        if (arr == NULL) {
            // allocation failed
            return;
        }

        // use arr here ...
        arr[0] = 42;

        free(arr);   // give memory back to the heap
    }
Takeaway: heap memory = manual. You ask for it, you free it. Forgetting free is a leak.
Heap bugs

Two classic mistakes when working with heap memory:

1. Memory leak

You take memory from the heap, but never give it back. 

    void leak_example(void) {
        int *arr = malloc(100 * sizeof(int));
        if (arr == NULL) return;

        arr[0] = 10;

        // BUG: forgot free(arr);
        // memory is still reserved on the heap
    }
2. Use-after-free

You keep using memory after you told the system you are done with it. 

    void use_after_free(void) {
        int *x = malloc(sizeof(int));
        *x = 5;
        free(x);          // memory is returned to the heap

        *x = 7;           // BUG: x points to memory we no longer own
    }
Takeaway: on the heap you can hurt yourself in slow and invisible ways (leaks, corrupt data). On the stack it usually crashes fast.

Same ideas apply in Zig or any low-level language. The important part is where the data lives, not the exact syntax.

Typical memory bugs
The stuff that makes programs crash or slowly die.
Bugs
Stack bugs
  • Stack overflow ‚Äì you use more stack than is available.
  • Often caused by deep recursion.
  • Also caused by huge local arrays.
  • Usually ends in a fast crash or kernel panic.
Heap bugs
  • Memory leak ‚Äì you never free what you allocated.
  • Use-after-free ‚Äì using memory after freeing it.
  • Double free ‚Äì freeing the same pointer twice.
  • These may not crash immediately, which makes them hard to find.
Idea: stack bugs are often loud and quick. Heap bugs are often quiet and slow. Both are dangerous in low-level code.
How to choose ‚Äì stack or heap?
A tiny decision checklist.
Decision
Ask these questions
  • How long does the data need to live?
  • How big can it get?
  • Do I know the size at compile time?
Choose stack if‚Ä¶
  • The data lives only inside one function.
  • The size is small and fixed.
  • You want speed and simplicity.
Choose heap if‚Ä¶
  • The data must live after the function returns.
  • The data can be large or variable size.
  • You are building shared structures (lists, trees, etc.).
Mental model: the stack is your short‚Äëterm scratchpad. The heap is your long‚Äëterm storage room. Both are in RAM, just used in different ways.
