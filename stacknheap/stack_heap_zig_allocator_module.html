Concept
Memory in a program

Your program gets some RAM. Two important regions you hear about:

  • Stack â€“ shortâ€‘lived data while functions run.
  • Heap â€“ flexible storage for longâ€‘lived or bigger data, controlled by an allocator.
The stack

Like a stack of plates:

  • You put new plates on top.
  • You always remove from the top.
  • Last in = first out (LIFO).

Zig uses the stack for:

  • Function parameters.
  • Local variables.
  • Return address.
The heap

The heap is a big pool of memory. You never touch it directly in Zig â€“ you go through an allocator.

  • You ask the allocator for memory (allocate).
  • You get back a slice or pointer.
  • You must free it when you are done.

In Zig, that allocator can be std.heap.GeneralPurposeAllocator, a fixed buffer, arena, etc.

Mental shortcut: stack = compiler manages it; heap = your allocator manages it. Same RAM, different rules.

Stack example

Normal function locals in Zig live on the stack. When the function returns, they disappear.

Local variables on the stack Stack 
const std = @import("std");

fn add(a: i32, b: i32) i32 {
    const sum = a + b; // sum lives on the stack
    return sum;        // a, b, sum are gone after return
}

pub fn main() !void {
    const result = add(2, 3); // result lives on the stack in main
    std.debug.print("result = {d}\n", .{result});
}
Allocator setup

This is the typical pattern to get a generalâ€‘purpose heap allocator in Zig. The allocator itself usually lives on the stack, but it hands out heap memory.

std.heap.GeneralPurposeAllocator Allocator 
const std = @import("std");

pub fn main() !void {
    var gpa = std.heap.GeneralPurposeAllocator(.{}){};
    defer _ = gpa.deinit(); // clean up internal bookkeeping

    const allocator = gpa.allocator(); // this is your heap interface

    // from here on, pass `allocator` to functions that need heap memory
}
Important: in Zig you rarely call a global malloc. You pass an allocator around so the caller decides where heap memory comes from.
Heap allocation example

Here we allocate an array on the heap using the allocator. The slice itself lives on the stack, but its backing memory is on the heap.

Allocating on the heap via allocator Heap 
const std = @import("std");

fn makeArray(allocator: std.mem.Allocator, count: usize)
    ![]u32 // slice backed by heap memory
{
    var slice = try allocator.alloc(u32, count); // heap allocation
    errdefer allocator.free(slice);             // free if we error later

    slice[0] = 42;

    return slice; // caller now owns the slice and must free it
}

pub fn main() !void {
    var gpa = std.heap.GeneralPurposeAllocator(.{}){};
    defer _ = gpa.deinit();
    const allocator = gpa.allocator();

    var data = try makeArray(allocator, 4);
    defer allocator.free(data); // give memory back

    std.debug.print("data[0] = {d}\n", .{data[0]});
}
Takeaway: allocator.alloc = get heap memory, allocator.free = give it back. Exactly the same idea as malloc/free, but wrapped in Zig style.
Typical heap bugs in Zig

The mistakes are the same as in C, just with nicer syntax.

1. Memory leak
fn leakExample(allocator: std.mem.Allocator) !void {
    const buf = try allocator.alloc(u8, 1024);
    // use buf...

    // BUG: forgot allocator.free(buf);
    // memory stays reserved for the rest of the program
}
2. Use-after-free
fn useAfterFree(allocator: std.mem.Allocator) !void {
    var slice = try allocator.alloc(u8, 4);
    slice[0] = 1;
    allocator.free(slice); // slice is now invalid

    slice[0] = 2; // BUG: writing through freed memory
}
Takeaway: allocators make heap use explicit, but they do not magically save you from leaks or useâ€‘afterâ€‘free. You still have to be disciplined.

You can plug in other allocators (arena, fixed buffer, page allocator) without changing the functions that only depend on std.mem.Allocator.

Stack vs heap â€“ bug overview
What usually goes wrong.
Bugs
Stack bugs
  • Stack overflow â€“ recursion or huge locals eat the stack.
  • Usually causes a quick crash or panic.
  • Fix by reducing depth or moving big things to heap.
Heap bugs
  • Memory leak â€“ you never free a slice.
  • Use-after-free â€“ you keep a slice after calling free.
  • Double free â€“ call free twice on the same memory.
Pattern: stack bugs are loud and fast, heap bugs are quiet and sneaky. Your allocator only makes the contract explicit â€“ it does not fix logic errors.
Choosing stack vs heap in Zig
Tiny checklist for real code.
Decision
Ask yourself
  • How long does the data need to live?
  • How big can it get?
  • Do I need to hand this data to some other part of the program?
Use stack ifâ€¦
  • The data lives only inside one function.
  • The size is small and fixed.
  • You want zero allocator overhead.
Use heap (allocator) ifâ€¦
  • The data must outlive the function.
  • The size is big or decided at runtime.
  • You want to share it between parts of the program.
Design tip: pass allocator: std.mem.Allocator into functions instead of hardâ€‘coding GeneralPurposeAllocator everywhere. That keeps your code flexible.
